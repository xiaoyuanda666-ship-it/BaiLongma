## 2. 详细问题分析

### 2.1 记忆系统问题 🔴

#### 问题 2.1.1：每次启动都重建 FTS 索引

**代码位置**：`src/db.js` 第 47 行
```javascript
db.exec(`INSERT INTO memories_fts(memories_fts) VALUES('rebuild')`)
```

**问题描述**：
- 每次启动都会执行 `rebuild` 操作
- 当记忆量大时，这会导致启动缓慢
- `rebuild` 是阻塞操作，影响启动时间

**影响**：
- 启动时间随记忆量线性增长
- 重启后需要等待索引重建完成

**建议**：
- 使用增量索引，或在非启动时后台执行
- 添加启动参数控制是否重建

#### 问题 2.1.2：记忆去重逻辑可能丢失重要变体

**代码位置**：`src/db.js` 第 95-120 行
```javascript
// 普通记忆去重：同类型且 content 前40字相同则跳过
const contentPrefix = (memory.content || '').slice(0, 40)
const dup = db.prepare(`
  SELECT id FROM memories WHERE event_type = ? AND content LIKE ? LIMIT 1
`).get(memory.event_type, `${contentPrefix}%`)
```

**问题描述**：
- 使用前 40 字判断重复，可能跳过语义不同但措辞相似的内容
- 同一个概念的细微差别可能被忽略

**影响**：
- 记忆的多样性降低
- 重要的上下文差异可能丢失

**建议**：
- 使用语义相似度而非字面匹配
- 引入向量嵌入进行相似度检测

#### 问题 2.1.3：记忆没有过期/归档机制

**问题描述**：
- 记忆永久存储，没有清理策略
- 随时间积累，数据库持续膨胀
- 没有冷热数据分离

**影响**：
- 长期运行后查询性能下降
- 存储成本增加

**建议**：
- 添加记忆 TTL 或访问频率过期
- 实现记忆摘要压缩
- 定期归档旧记忆

### 2.2 并发与竞态条件 🔴

#### 问题 2.2.1：处理标志不是真正的互斥锁

**代码位置**：`src/index.js` 第 155 行
```javascript
let processing = false

async function onTick() {
  if (processing) return
  processing = true
  // ...
}
```

**问题描述**：
- 使用简单布尔标志而非真正的互斥锁
- 在极高负载下可能仍会出现竞态
- `hasMessages()` 和 `popMessage()` 之间没有原子性保障

**影响**：
- 极端情况下消息可能被重复处理或丢失

**建议**：
- 使用 `Atomics` 或 `Mutex` 实现真正互斥
- 或使用消息队列中间件

#### 问题 2.2.2：队列没有持久化

**代码位置**：`src/queue.js` 第 5 行
```javascript
const queue = []
```

**问题描述**：
- 队列完全存储在内存中
- 进程异常退出时消息丢失

**影响**：
- 消息可靠性无法保证
- 系统重启后未处理消息丢失

**建议**：
- 使用 Redis 或其他持久化队列
- 或在 SQLite 中存储待处理消息

#### 问题 2.2.3：AbortController 的时序问题

**代码位置**：`src/index.js` 第 100 行
```javascript
currentAbortController = new AbortController()
```

**问题描述**：
- 新消息到达时触发中断，但可能中断正在执行工具调用的循环
- 工具执行到一半被中断可能留下不一致状态

**影响**：
- 部分工具调用已执行但结果未处理
- 可能产生副作用

**建议**：
- 添加事务日志记录工具调用进度
- 实现补偿机制处理中断的调用

### 2.3 配额管理系统 🟡

#### 问题 2.3.1：每日配额是内存状态

**代码位置**：`src/quota.js` 第 70 行
```javascript
const dailyUsage = {}   // { capability: { date: 'YYYY-MM-DD', count: number } }
```

**问题描述**：
- 每日配额存储在内存中
- 重启后配额计数重置
- 没有利用每天的可用配额

**影响**：
- 重启后浪费已购配额
- 无法准确追踪实际消耗

**建议**：
- 将配额使用记录持久化到 SQLite
- 跨重启累计配额消耗

#### 问题 2.3.2：固定限流值不灵活

**代码位置**：`src/quota.js` 第 9 行
```javascript
const LIMITS = {
  RPM: 500,
  TPM: 20_000_000,
}
```

**问题描述**：
- 硬编码 RPM/TPM 限制
- 不同 API Tier 有不同限制
- 无法适配不同的账户层级

**影响**：
- 低配额账户被限制，高配额账户浪费
- 无法动态调整

**建议**：
- 从 API 响应中动态获取配额限制
- 支持配置不同的 Tier

#### 问题 2.3.3：滑动窗口可能遗漏突发流量

**代码位置**：`src/quota.js` 第 18 行
```javascript
const WINDOW_MS = 60 * 1000
```

**问题描述**：
- 60 秒窗口可能无法捕捉短时间突发
- 某些 API 按秒限流

**影响**：
- 可能触发 API 的秒级限流
- 不够精细的限流控制

**建议**：
- 多窗口追踪（60s + 10s）
- 结合 API 返回的限流信息

### 2.4 LLM 层可靠性 🟡

#### 问题 2.4.1：工具调用轮数过高

**代码位置**：`src/llm.js` 第 130 行
```javascript
const MAX_TOOL_ROUNDS = 10
```

**问题描述**：
- 最多 10 轮工具调用（injector 里写 12，llm.js 里写 10）
- 每轮可能产生多个工具调用
- 组合起来可能产生大量 API 调用

**影响**：
- 单次响应成本不可控
- 可能陷入工具调用的无限循环

**建议**：
- 降低轮数限制
- 添加每轮 token 预算
- 实现智能循环检测

#### 问题 2.4.2：XML 解析是脆弱的 fallback

**代码位置**：`src/llm.js` 第 66 行
```javascript
// 从内容中解析 XML 格式的工具调用（MiniMax 有时输出 XML 而非 JSON tool_calls）
function parseXmlToolCalls(content) {
```

**问题描述**：
- XML 解析是备用方案，不是可靠设计
- 依赖模型输出特定格式
- 没有验证解析结果的正确性

**影响**：
- 模型输出格式变化会导致解析失败
- 可能执行错误的工具调用

**建议**：
- 使用官方的 function calling 接口
- 而不是依赖文本解析

#### 问题 2.4.3：没有降级策略

**问题描述**：
- 没有备用模型列表
- API 失败时没有降级方案

**影响**：
- API 不可用时系统完全停止

**建议**：
- 实现多模型/多 API Key 降级
- 添加Circuit Breaker模式

### 2.5 安全边界问题 🟡

#### 问题 2.5.1：保护文件列表过小

**代码位置**：`src/capabilities/executor.js` 第 60 行
```javascript
const PROTECTED_FILES = new Set(['readme.txt', 'world.txt', 'package.json'])
```

**问题描述**：
- 只保护 3 个文件
- 没有目录保护
- `package.json` 在保护列表但不在沙盒

**影响**：
- 可能误删重要文件
- 没有目录级别的保护

**建议**：
- 扩展保护列表
- 支持目录级保护规则

#### 问题 2.5.2：exec_command 风险

**代码位置**：`src/capabilities/executor.js` 第 135 行
```javascript
async function execCommand(args) {
```

**问题描述**：
- 在沙盒内执行任意 shell 命令
- 即使在沙盒内，仍可能有风险
- 没有命令白名单

**影响**：
- 可能执行恶意命令
- 资源耗尽攻击

**建议**：
- 添加命令白名单
- 添加执行超时和资源限制
- 记录所有命令执行

#### 问题 2.5.3：fetch_url 没有内容过滤

**代码位置**：`src/capabilities/executor.js` 第 216 行
```javascript
async function execFetchUrl(args) {
```

**问题描述**：
- 抓取任意 URL 的内容
- 没有内容大小/类型限制
- 没有恶意内容检测

**影响**：
- 可能抓取恶意内容
- 可能触发 SSRF 攻击

**建议**：
- 添加 URL 白名单/黑名单
- 内容大小和类型限制
- 添加恶意内容检测

### 2.6 可观测性不足 🟡

#### 问题 2.6.1：日志不够结构化

**问题描述**：
- 使用 `console.log` 输出
- 没有日志级别
- 没有请求追踪 ID

**影响**：
- 难以排查问题
- 无法关联日志

**建议**：
- 使用结构化日志库
- 添加请求追踪 ID
- 支持日志级别配置

#### 问题 2.6.2：没有健康检查端点

**代码位置**：`src/api.js`

**问题描述**：
- 只有 `/status` 端点
- 没有深度健康检查
- 没有依赖检查（数据库、API 连接）

**影响**：
- 无法判断系统健康状态
- 负载均衡无法正确判断

**建议**：
- 添加 `/health` 端点
- 检查所有依赖的可用性

#### 问题 2.6.3：没有指标收集

**问题描述**：
- 没有 Prometheus/Metrics 端点
- 无法监控系统性能
- 无法告警

**影响**：
- 运维困难
- 问题被动发现

**建议**：
- 添加指标收集
- 导出 Prometheus 格式

### 2.7 可扩展性限制 🔴

#### 问题 2.7.1：单实例架构

**问题描述**：
- 只能单实例运行
- 无法水平扩展
- 没有负载均衡

**影响**：
- 无法处理高并发
- 单点故障

**建议**：
- 支持多实例部署
- 添加消息中间件
- 实现状态外部化

#### 问题 2.7.2：没有分布式支持

**问题描述**：
- 所有状态在本地
- 没有分布式锁
- 没有跨实例消息

**影响**：
- 多实例会有状态不一致

**建议**：
- 使用分布式状态存储
- 实现分布式锁

## 3. 其他设计问题

### 3.1 提示词依赖外部模型能力

**代码位置**：`src/prompt.js`

**问题描述**：
- 系统提示词依赖模型遵循复杂指令
- `[SET_TASK]`、`[RECALL]` 等标记依赖模型正确输出
- 没有解析失败的回退

**影响**：
- 模型输出格式不稳定会导致系统行为异常

### 3.2 上下文采集器可能无限循环

**代码位置**：`src/context/gatherer.js`

**问题描述**：
- 虽然有 `MAX_ROUNDS = 3` 限制
- 但检查器返回 `sufficient: false` 时可能继续
- 每轮调用 LLM，成本累积

**影响**：
- 任务模式下可能产生额外 API 成本

### 3.3 实体册没有清理

**代码位置**：`src/db.js`

**问题描述**：
- `entities` 表只增不减
- 没有实体过期或清理机制

**影响**：
- 长期运行后实体表膨胀

## 4. 问题优先级建议

### 高优先级（应该立即修复）

| 问题 | 修复方案 |
|------|----------|
| 队列持久化 | 消息写入 SQLite 或使用 Redis |
| 每日配额持久化 | 配额记录写入 SQLite |
| 添加健康检查 | 实现 `/health` 端点 |
| exec_command 限制 | 命令白名单 + 超时限制 |

### 中优先级（建议修复）

| 问题 | 修复方案 |
|------|----------|
| 记忆过期机制 | 添加 TTL 或访问频率过期 |
| 工具调用轮数限制 | 降低轮数 + token 预算 |
| 结构化日志 | 使用日志库添加追踪 ID |
| 每日配额跨重启 | 持久化配额使用记录 |

### 低优先级（长期改进）

| 问题 | 修复方案 |
|------|----------|
| 多实例支持 | 状态外部化 + 分布式架构 |
| 分布式锁 | 使用分布式锁服务 |
| 指标收集 | 添加 Prometheus 导出 |
| 记忆摘要压缩 | 实现记忆压缩/归档 |

---

## 5. 总结

当前架构存在以下核心问题：

1. **可靠性不足**：内存状态导致重启后状态丢失，消息队列不可靠
2. **安全边界模糊**：保护文件列表小，exec_command 风险高
3. **可扩展性差**：单实例设计，无法水平扩展
4. **可观测性弱**：日志不结构化，没有指标
5. **长期稳定性**：记忆无过期机制，数据库会持续膨胀

建议优先修复高优先级问题，确保系统在生产环境中的可靠性。
